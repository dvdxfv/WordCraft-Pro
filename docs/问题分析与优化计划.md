### ✅ 问题1 — AI 链接配置

**问题描述**：前端设置面板未提供 AI 服务配置入口，用户无法自行切换模型或填写 API Key。

**根本原因**：AI 调用硬编码在 `app.py:callAI`，通过两条通道发出请求：
1. Supabase Edge Function 代理（`/functions/v1/ai-proxy`，豆包 Seed 1.6）
2. 直连豆包 ARK API（`ark.cn-beijing.volces.com`，模型 `doubao-seed-1-6-251015`）

**修复内容**（前次会话）：
- 设置面板已有模型选择下拉（豆包系列）及"测试连接"按钮
- `callAI` 双通道（Supabase Edge → 直连豆包），任意一条通畅即可工作
- `config.yaml` 中 `llm.api` 段落可切换 provider / model / api_key

### ✅ 问题2 — 导出文件乱码（2026-04-20）

**问题描述**：点击"导出"按钮生成的 `.docx` 文件在 Word 中打开时，中文字符显示为乱码或方框。

**根本原因**：前端 `exportDoc()` 使用 `docx.js` 库生成 `.docx`，但 h2、h3、ref/caption、正文段落的 `TextRun` 均未设置东亚字体（`eastAsia`），Word 回落到内建默认字体 Calibri，Calibri 不支持中文，导致乱码。

h1 虽然设置了 `font.name`，但写的是中文字体名（如 `黑体`），西文字符也会用黑体渲染，风格不一致。

**修复文件**：`web/index.html:exportDoc()`

**修复内容**：
- 新增读取 `fH2Font`/`fH2Size`、`fH3Font`/`fH3Size`、`fWFont`（西文字体）共 5 个表单值
- 所有 `TextRun` 统一设置 `font: { name: wFont, eastAsia: cnFont }`
  - `name`（ASCII/西文）→ `fWFont`（默认 Times New Roman）
  - `eastAsia`（中文）→ 对应级别的中文字体
- 新增 h3 类型显式处理分支（`HeadingLevel.HEADING_3`）
- ref、caption 类型同步加上字体设置

**验证方法**：导出包含中文标题和正文的 `.docx`，在 Word 中打开，各级标题和正文字体应与面板设置一致。

### ✅ 问题3 — 排版规则上传/AI解析失败（2026-04-20）

**问题描述**：
1. 在 AI 排版面板上传参考模板（.docx/.pdf）时，浏览器控制台报 `Maximum call stack size exceeded`，上传失败
2. 点击"AI 解析排版规则"按钮无响应，仅弹出 `'AI 解析中…（需配置 API Key）'` toast，功能为桩函数
3. 点击"应用到文档"同样是桩函数，仅弹出 `'已应用 AI 规则'`

**根本原因**：
- **上传崩溃**：`handleTemplateUpload` 使用 `btoa(String.fromCharCode(...new Uint8Array(buf)))` 将整个文件通过展开运算符传入，JavaScript 调用栈约 65 000 帧，典型 .docx 文件远超此限
- **功能缺失**：`runAIParse` 和 `applyAIRules` 从未实现

**修复文件**：`web/index.html`

**修复内容**：

*1. 修复上传崩溃（`handleTemplateUpload`）*
- 改为 8 192 字节分块编码：`String.fromCharCode(...bytes.subarray(i, i+chunk))`
- 上传成功后仅在 `aiInput` 为空时写入提示文案，不覆盖用户已有内容

*2. 实现 `runAIParse`*
- 读取 `aiInput` 中的自然语言需求
- 调用 `WC_API.callAI` 发送给豆包，系统 Prompt 要求返回固定键名的 JSON
- 从响应中提取 JSON（`reply.match(/\{[\s\S]*\}/)`），解析后存入 `window._aiParsedRules`
- 在 `#aiRulesBody` 中渲染键值列表（含中文标签）
- 按钮（`id="btnAIParse"`）在请求期间 `disabled + opacity:0.6`，完成后恢复

*3. 实现 `applyAIRules`*
- 读取 `window._aiParsedRules`，将对应键值填入所有表单字段：
  `fH1Font`, `fH1Size`, `fH2Font`, `fH2Size`, `fH3Font`, `fH3Size`, `fBFont`, `fBSize`, `fWFont`, `fIndent`, `fLH`, `fAlign`, `fMT`, `fMB`, `fML`, `fMR`

**同步修复（审查后）**：
- AI 解析按钮 `id` 从 CSS 类路径改为 `id="btnAIParse"`，消除脆弱选择器

### ✅ 问题4 — 仪表板数据未更新（2026-04-20）

**问题描述**：切换文件标签时，右侧 QA 面板和交叉引用面板数据被清空，再切回也不会恢复，导致必须重新运行检查。

**根本原因**：`switchTab()` 调用渲染函数时未区分"清空"和"恢复已有结果"，所有 tab 共享同一套面板变量，切换时直接归零。

**修复文件**：`web/index.html`

**修复内容**：
- 新增 `tabQAState`、`tabXrefState` 两个 Map（以文件名为 key）
- 新增 `saveTabDashboardState()`：离开 tab 前保存当前面板数据到 Map
- 新增 `restoreTabDashboardState(fn)`：进入 tab 时从 Map 恢复（有缓存则渲染，无则显示空状态）
- `switchTab()` 改为先 save 后 restore，不再一律清空
- `runQA()` 和 `runXRef()` 完成后自动写入各自 State Map

### ✅ 问题6 — 图片处理（2026-04-20）

**问题描述**：
1. 解析 `.docx` 时图片未被提取，内嵌图片在预览中不显示
2. 原文档中居中的图片，在应用预览中左对齐
3. 无法对图片进行对齐操作

**根本原因（后端）**：`DocxParser._extract_images()` 仅扫描文档关系（`.rels`），未实际遍历段落中的 `<w:drawing>` 元素，导致内嵌图片漏检；也未读取段落对齐属性。

**修复文件**：`parsers/docx_parser.py`、`web/index.html`

**修复内容**：

*后端（图片识别）*
- `_extract_images()` 改为扫描所有段落的 `<w:drawing>` 元素
- 从 `<wp:extent cx cy>` 提取宽高（EMU → cm）
- 从 `<a:blip r:embed>` 获取 rel_id，映射到实际文件路径
- 读取段落 `<w:jc>` 属性，写入 `ImageData.alignment`

*前端（对齐 CSS）*
- 新增 `.img-center`、`.img-left`、`.img-right` 样式类
- 新增 docx-preview 渲染段落对齐 → 图片对齐的 CSS 映射规则

*前端（操作工具栏）*
- 点击 `docPage` 内任意图片，弹出浮动工具栏（绝对定位，图片正上方）
- 工具栏提供左对齐 / 居中 / 右对齐三个按钮，当前对齐状态高亮
- 点击文档其他区域自动关闭工具栏

### ✅ 问题7 — .doc 测试文件不显示（2026-04-20）

**问题描述**：打开 `samples/益海嘉里项目综合效益评价（打印最终版）.doc` 时，浏览器长时间无响应，最终提示"打开失败"或静默失败。

**根本原因（两个独立 Bug）**：

1. **前端 UI 冻结**：`_loadDocFile` 用逐字符循环做 base64 编码：
   ```js
   let bin=''; for(let i=0;i<bytes.length;i++) bin+=String.fromCharCode(bytes[i]);
   ```
   字符串拼接时间复杂度为 O(n²)，该测试文件约 44 MB，循环 4 400 万次，浏览器主线程冻结数十秒。

2. **后端单点依赖**：`_convert_doc_to_docx` 只支持 LibreOffice，未安装时直接 `raise RuntimeError`，且无友好提示。

**修复文件**：`web/index.html`、`parsers/dispatcher.py`

**修复内容**：

*前端（`_loadDocFile`）*
- 同问题3的分块方案：8 192 字节分块，`String.fromCharCode(...bytes.subarray(i, i+chunk))`
- toast 提示文案改为"`.doc` 转换中，请稍候…"

*后端（`_convert_doc_to_docx`）—— 三级回退*

| 优先级 | 方法 | 依赖 |
|--------|------|------|
| 1 | python-docx 直接打开（部分 .doc 实为 Open XML） | 仅 python-docx |
| 2 | LibreOffice headless 转换 | LibreOffice 安装 |
| 3 | Windows PowerShell + Word COM | Microsoft Word 安装 |

- LibreOffice 转换失败时改为 `warning + fall through`（不再 `raise`），确保 Word COM 回退能被触发
- 三级均失败时抛出清晰错误，注明两种安装方案
- 新增 `.fi-icon.doc` / `.ft-icon.doc` CSS，文件树和标签页的图标颜色正确显示

### ⏳ 问题8 — Supabase 架构优化（长期可选）

**问题描述**：当前 Supabase 仅作为数据库和用户认证层使用，未利用 Edge Functions 处理 AI 代理和核心业务逻辑，所有请求都绕道本地 Flask，增加部署复杂度。

**现状**：
- 用户认证：Supabase Auth（邮箱登录）
- 数据存储：PostgreSQL（用户模板、文档、设置）
- 文件存储：Supabase Storage（模板文件）
- AI 代理：Supabase Edge Function（`/functions/v1/ai-proxy`，已部署，作为 `callAI` 通道1）

**优化方向**：
- 将更多后端逻辑（排版规则解析、QA 摘要等轻量任务）迁移至 Edge Functions
- 前端直接调用 Supabase，减少对本地 Flask 的依赖
- 利用 Supabase Realtime 实现云文档实时同步

**优先级**：低，当前 Flask + Supabase 混合架构已能正常运行，等核心功能稳定后再考虑。
