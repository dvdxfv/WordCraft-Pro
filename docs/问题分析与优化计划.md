### ✅ 问题2 — 文档预览背景色不统一

**修复日期**：2026-04-21

**症状**：打开"毕业论文终稿.docx"时预览区显示为黑底白字，与其他文档（白纸 + 灰面板 + 深色视窗）风格不一致。

**原因**：docx-preview 会把 docx 内部主题色写入 `section.docx` 的 inline `background-color`（以及段落级 inline 背景色）。我们的 `.doc-page.is-docx ... >section.docx { background: var(--bg2) }` 规则特异度不够，被 inline 样式或 docx-preview 生成的 scoped `<style>` 覆盖。

**修复**（`web/index.html` 预览区 CSS）：
- `section.docx` 的 `background` / `color` 改用 `!important`，彻底压过 inline 和 scoped 样式
- 深色主题同步加 `!important`
- 追加规则：`.doc-page.is-docx .docx-wrapper [style*="background"] { background: transparent !important }` —— 统一清除段落/表格内联的背景色，防止"整页白纸上的深色色块"

### ✅ 问题4 — 模板上传后显示 null 值

**修复日期**：2026-04-22

**根本原因**：`uploadTemplate` 后端只存储文件，不提取格式；前端 `handleTemplateUpload` 将提示语写入 `aiInput`（"已上传模板：xxx.docx，请点击 AI 解析…"），`runAIParse` 把这句话送给 AI，AI 找不到任何格式信息，对所有字段返回 `null`。

**修复**（两处）：

| 层 | 修改 | 作用 |
|---|---|---|
| 后端 `app.py` | 新增 `_extract_docx_format_rules(file_path)` 静态方法 | 用 `python-docx` 扫描前 300 段，提取标题 1~3 字体/字号、正文字体/字号/西文字体/首行缩进/行距/对齐、四边页边距；从 XML `w:rFonts[@w:eastAsia]` 读中文字体名，比 `run.font.name`（西文名）更准确 |
| 后端 `app.py:uploadTemplate` | `.docx` 文件上传后调用上述方法，在响应中附加 `format_rules` 字段 | 不论是 Supabase 模式还是本地模式都返回提取结果 |
| 前端 `web/index.html:handleTemplateUpload` | 检测 `data.format_rules`：若有内容直接设置 `window._aiParsedRules` 并渲染 `#aiRulesBody`，同时弹 toast 提示"点击应用到文档"；若无内容（如 PDF 模板）则回退到旧提示语行为 | 消除 AI 中间环节，直接展示真实提取值 |

**附注**：
- PDF 模板无法用 `python-docx` 解析，仍走旧流程（显示提示语，用户手动输入描述再 AI 解析）
- 提取到的规则已设置到 `window._aiParsedRules`，可直接点"应用到文档"填入格式面板，无需再点"AI 解析排版规则"

### ✅ 问题5 — 导出文档格式混乱

**修复日期**：2026-04-21（分两批）

#### 第一批修复（乱码 `<a>` 标签）

**原因**：`docContents[name]` 元素的 `.text` 字段存的是 innerHTML（含 `<a id="_Toc...">` 等标签），直接传给 docx.js TextRun 导致乱码。

**修复**：新增 `_strip(s)` 辅助函数，用 DOMParser 剥离 HTML 标签，TextRun 只传纯文本。

#### 第二批修复（排版全乱）

**原因**：首行缩进系数 `0.35` 应为 `0.035`（1pt≈0.035cm），导致所有正文段落首行缩进约 3.3 英寸（10 倍过大），排版完全错乱。同时缺少页面设置和列表项处理。

**修复**：
- **首行缩进**：修正系数 `0.35` → `0.035`；计算公式：`indent_chars × bSize_pt × 0.035 ÷ 2.54 × 1440 twips`
- **页面设置**：Document sections 加 A4 页面（11906×16838 twips）和标准页边距（上下 2.54cm、左右 3cm）
- **列表项**：新增 `li` 类型分支，渲染为 `• ` 前缀 + `indent.left:480` 段落（原来回退为正文段落）
- **标题间距**：h1/h2/h3 段落加 `spacing.before`/`spacing.after`（h1: 240/120，h2: 200/100，h3: 160/80 twips）

#### 第三批进展（自动化回归测试）

**完成日期**：2026-04-21

**新增测试**：
- 新建 `tests/e2e/test_export_docx_regression.py`（E2E）：
  1. 自动拉起 `web/flask_app.py`（5000）与 `web/run_web.py`（8081）
  2. 打开 `web/login.html` 登录后进入 `index.html`
  3. 上传 `samples/南海鸢乌贼捕捞量智能反演文献综述_导出.docx`
  4. 执行导出并捕获下载文件
  5. 用 `python-docx` 对原文档与导出文档做排版签名比对（section/段落/缩进/间距/run 字体等）
- 新增 `pytest.ini`，注册 `e2e` marker

**初次回归结论（第三批）**：❌ 未通过——发现页边距/首行缩进/段前后距三类差异，产出 `tests/artifacts/phase5_export_compare_report.json` 作为二次修复基线。

#### 第四批修复（二次修复，E2E 回归通过）

**修复日期**：2026-04-22

按 `tests/artifacts/phase5_export_compare_report.json` 明细逐项修复，`tests/e2e/test_export_docx_regression.py` 回归测试已全部通过。

### ✅ 问题6 — QA 质量检查依赖 AI

**修复日期**：2026-04-21

**根本原因（两个独立问题）**：

1. **前端 `runQA()` 根本没调后端** —— 该函数只读取 `fileQaData` 这个硬编码的 mock（仅 3 个示例文件），换成任意真实文档点"规则检查"全部返回 0 条问题，给用户"只能靠 AI"的错觉。
2. **后端规则覆盖面有限** —— 已有 `typo_checker` / `consistency_checker` / `logic_checker` / `crossref_checker`，但没有标点、空格、括号、重复字符等浅层格式检查，实际写作中这些才是最常见、最容易被肉眼漏掉的问题。

**修复**：

| 层 | 修改 | 作用 |
| --- | --- | --- |
| 后端 | 新增 `core/punctuation_checker.py`（`PunctuationChecker`） | 6 类规则：中英文标点混用、连续重复标点、中英文间缺空格、全角数字、括号/引号配对、连续重复汉字 |
| 后端 | `core/qa_engine.py` 注册 `PunctuationChecker`，新增 `format` 类别，默认类别补入 | 打通到 `QAEngine.check()` |
| 后端 | `app.py:runQA` 默认 `categories` 增加 `"format"` | 不传类别时也会执行标点检查 |
| 前端 | `web/index.html:runQA()` 改为 `async` 函数，调用 `WC_API.runQA(content, categories)` | 真正运行后端规则检查 |
| 前端 | 字段映射：`severity warning→warn`、`category typo→错别字 / inconsistency→数据不一致 / format→格式问题 / logic→逻辑问题`；`location_text→hlText`；按 severity 推导 `hlClass` | 后端问题结构 → 前端 `qaData` 结构 |
| 前端 | 后端失败时回退到 `fileQaData` 的本地示例，保证 UI 不崩 | 兼容离线 |

**新增的标点规则细节**：

| 规则 | 置信度 | 严重级 | 举例 |
| --- | --- | --- | --- |
| 中英文标点混用 | 0.85 | warning | `中文段落,` → `中文段落，` |
| 连续重复标点 | 0.9 | warning | `！！` / `。。` / `,,` |
| 中英文间缺空格 | 0.4 | info | `使用Python` → `使用 Python`（每段最多 3 条防刷屏） |
| 全角数字 | 0.95 | warning | `２０２５` → `2025` |
| 括号不配对 | 0.85 | warning | `（…` 缺 `）` |
| 连续 3+ 相同汉字 | 0.95 | error | `的的的` |
| 连续 2 个相同汉字（非叠字白名单） | 0.35 | info | `的的`、`了了`（白名单收录 50+ 合法叠字，如"渐渐""慢慢""爸爸"） |

**后续增强（2026-04-21 追加）**：扩充错别字词库，由 158 条 → **~580 条**：

| 改动 | 说明 |
| --- | --- |
| 新增 `core/data/common_typos.tsv` | 基于《现代汉语常见错别字手册》《咬文嚼字》编者汇编、公文与学术写作常见错误，手工整理 ~550 条 TSV |
| 新增 `core/data/sighan_pairs.txt` | 基于 [shibing624/pycorrector](https://github.com/shibing624/pycorrector)（Apache-2.0）的 SIGHAN 2015 语料衍生 347 条 2-gram 对，opt-in 加载（默认关闭，噪声高） |
| 重构 `core/typo_lib.py` | 统一加载：手工词典 + common_typos.tsv + user_typos.tsv + （可选）SIGHAN；自动过滤单字 / 自映射 / 建议含"/"等噪声条目 |
| 用户扩展 | 支持 `core/data/user_typos.tsv`（已加入 `.gitignore`），用户可覆盖或追加项目专有词条 |
| 顺带修复 | 旧版 CONFUSING_WORDS 中单字条目（如 `"的":("的/地/得",...)`）导致 TypoChecker 为每个"的""地""得"刷屏的误报，已随新过滤器自动剔除 |

### ✅ 问题7 — 采纳修改功能不同步

**修复日期**：2026-04-21

**症状**：
1. 在文章中点击"采纳"有修改效果
2. 但在右边工作区点击"采纳"无反应
3. 文章中的采纳已经点击了，但右边工作区状态未同步更新

**根本原因（4 个独立 bug）**：

| Bug | 位置 | 原因 |
| --- | ---- | ---- |
| 文章采纳不同步右侧面板 | `acceptSuggestion` | 只改 DOM，从不更新 `currentQAData[i].status` 或卡片 class |
| 右侧面板采纳无反应 | `acceptIssue` | 调用后端 `acceptSuggestion`；文章已采纳后 `original === suggestion`，后端返回 `success:false`；且 `page.innerHTML = corrected_content` 销毁所有高亮元素的事件监听 |
| 状态未同步（已采纳仍显示 pending） | `acceptSuggestion` | 同第 1 个 bug |
| 切标签后高亮 index 错位 | `applyHighlights` / `restoreTabDashboardState` | 传入过滤后的 `pending` 数组，`idx` 不等于原始 issue 编号 |

**修复**：
- 新增 `_markIssueAccepted(i)` 共享函数，统一更新状态、卡片、toast、`checkAllDone()`
- `acceptSuggestion`（文章内）：采纳后读 `hl.dataset.issueId`，调用 `_markIssueAccepted(i)`
- `acceptIssue`（右侧面板）：废弃后端调用，改为纯本地 DOM 操作，同样调用 `_markIssueAccepted(i)`
- `applyHighlights`：用 `item.id`（QA 运行时赋值的原始 index）而非循环 `idx` 作为 `data-issue-id`；跳过非 `pending` 条目
- `restoreTabDashboardState`：改传全量 `qa` 数组（而非过滤后的 `pending`）给 `applyHighlights`
