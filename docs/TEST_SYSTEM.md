# 测试体系

这份文档定义仓库的正式测试标准，不是“建议做一下测试”。

## 目标

测试体系要解决两件事：

1. 历史上已经修过的问题，不能悄悄回来。
2. 新批次在写计划时，必须先把“具体怎么验收”写清楚，不能靠事后补口头说明。

## 分层

### 1. 核心回归闸门

正常改动后，至少要跑下面三组：

```bash
python -m pytest tests/test_batch_regression.py -v
python -m pytest tests/test_format_checker.py -v
python -m pytest tests/e2e/test_batch7_e2e.py -m "e2e and no_login" -v
```

作用：

- `test_batch_regression.py`：保护后端和规则层的历史回归点
- `test_format_checker.py`：保护格式检查核心逻辑
- `test_batch7_e2e.py -m "e2e and no_login"`：保护关键前端 JS 行为，而且默认不依赖真实账号

规则：

- 前端行为改动，不能只靠后端测试兜底。
- 如果改动范围超出这三组，当前批次必须新增至少一条对应功能的自动化回归。

### 2. 功能回归

每个历史 P0 / P1 问题，最终都要落到某一种自动化测试里：

- 单元回归
- 集成回归
- 无登录浏览器回归
- 需登录浏览器回归
- 真实样本 smoke 回归

不能只写“人工测过”。只要问题可以再次由代码引入，就必须有自动化归属。

### 3. 真实样本 Smoke

Smoke 不是按 pytest 文件名组织，而是按“真实用户功能链”组织。

当前真实样本根目录：

`G:\开发项目\备份\samples`

当前样本清单：

`samples/manifest.json`

当前可执行命令：

```bash
python -m pytest tests/test_real_sample_smoke.py -m smoke -v
python tests/scripts/run_real_sample_smoke.py
python -m pytest tests/e2e/test_ai_parse_template_baseline.py -m "e2e and no_login" -v
```

Smoke 至少要覆盖这些功能链：

- 打开文件 -> 预览
- 规则 QA
- AI Parse / AI QA
- 交叉引用
- 格式合规
- 导出 / roundtrip

## 测试设计规则

### 行为优先

测试要写“用户看到的行为”，不要只写实现细节。

正确示例：

- 采纳错别字建议后，正文高亮文字被替换
- 采纳可直接替换的说明类建议后，正文高亮文字被替换
- 采纳解释型建议后，正文文字不应被误写
- 导入模板后，AI Parse 结果要能和标准答案逐项对照

错误示例：

- 函数返回了 truthy object
- 路径执行时没有抛异常
- 非错别字建议一律只标记采纳

### 默认可跑

仓库必须始终保留一条“不用真实账号也能跑”的浏览器回归线。登录态 E2E 可以有，但只能是补充，不能是唯一的前端保护。

### 历史问题入库

每修掉一个真实问题，都要做三件事：

1. 找到最小、稳定、可自动化的断言
2. 放进现有最接近的回归文件，或者新建合适的回归文件
3. 在对应批次计划里写明这条保护

### 单条测试职责要窄

每条回归最好只回答一个小问题。不要让一个超大测试同时承担太多不相关行为，否则坏了以后很难定位。

## AI Parse 专项标准

AI Parse 不能只测“返回了一个 JSON”。

标准流程必须贴近人为操作：

1. 导入真实模板
2. 查看导入后得到的模板文本
3. 运行 AI Parse
4. 将解析结果与对应标准答案逐项对照
5. 对不上就算失败

当前基线命令：

```bash
python -m pytest tests/e2e/test_ai_parse_template_baseline.py -m "e2e and no_login" -v
```

这条基线的意义是：至少保证“模板导入 -> 解析结果展示 -> 与答案对照”的验收口径被正式写进自动化，而不是只做主观判断。

## 计划文件要求

任何 `PLANS/` 下、文件名含 `plan` 的活动计划，都必须包含 `## 测试策略` 段落，并带这四个小节：

- `### 核心闸门`
- `### 新增回归覆盖`
- `### Smoke / 真实样本验证`
- `### 残余风险`

而且必须写具体：

- 尽量列出确切命令
- 说明本次保护的是哪类历史问题
- 说明还有哪些部分仍是人工验证或尚未自动化

只写“跑一下回归”“做一下人工验证”不算合格。

## 完成标准

一个批次不能只因为“代码改完了”就关闭。至少要同时满足：

1. 代码修复已经落地
2. 对应回归测试已新增或已更新
3. 核心回归闸门已执行
4. 需要真实样本时，已声明 smoke / 样本验证范围
5. 计划文件已经写清楚本次测试覆盖和残余风险
